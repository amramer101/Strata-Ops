## 🎯 What Is This Phase?

This phase takes the same 5-tier VProfile Java application from the previous phases and **containerizes it entirely using Docker Compose**, then automates the full deployment to a fresh AWS EC2 instance using **Ansible** — no manual SSH, no manual commands, no guesswork.

The key evolution here over Phase 2 & 3:

| Concern | Phase 2 (Lift & Shift) | This Phase (Docker Lift & Shift) |
|---|---|---|
| **Deployment Unit** | EC2 VMs (5 instances) | Docker Containers (1 instance) |
| **Config Management** | Manual + Terraform | Ansible Playbook |
| **Service Discovery** | Route53 Private DNS | Docker internal network |
| **Cost** | ~$70/month (5x EC2) | ~$20/month (1x t2.medium) |
| **Startup Time** | ~10 min (Terraform only) | ~8 min (Terraform + Ansible) |

### The Two-File Strategy (Dev vs. Prod)

```
docker-compose.yml          ← Development: builds images from source
  └── vprodb:  build: context: ../../
  └── vproapp: build: dockerfile: tomcat/Dockerfile
  └── vproweb: build: context: ./Docker-files/nginx

docker-compose.prod.yml     ← Production: pulls pre-built images from Docker Hub
  └── vprodb:  image: amrmamer/vprofiledb   ✅ No build needed on server
  └── vproapp: image: amrmamer/vprofileapp  ✅ Source code NOT required
  └── vproweb: image: amrmamer/vprofileweb  ✅ Ansible copies this file only
```

> 💡 **The key insight:** The production server has no source code and no need to compile anything. Ansible ships the `docker-compose.prod.yml` and `.env` file only, and Docker pulls everything else from Docker Hub.

## 🔨 The Dockerfiles — How Each Image Is Built

### 1. Tomcat App — Multi-Stage Build

```dockerfile
# Stage 1: Build the WAR file with Maven
FROM maven:3.9.9-eclipse-temurin-21-jammy AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline      # Cache dependencies layer
COPY src ./src
RUN mvn clean package -DskipTests  # Produces vprofile-v2.war

# Stage 2: Run the WAR on Tomcat (no Maven, no source code)
FROM tomcat:10-jdk21
RUN rm -rf /usr/local/tomcat/webapps/*
COPY --from=builder /app/target/vprofile-v2.war /usr/local/tomcat/webapps/ROOT.war
EXPOSE 8080
CMD ["catalina.sh", "run"]
```

### 2. MySQL DB — Pre-seeded Schema

```dockerfile
FROM mysql:8.0.33
ENV MYSQL_ROOT_PASSWORD=vprodbpass
ENV MYSQL_DATABASE=accounts
# Auto-runs on first container start:
COPY src/main/resources/db_backup.sql /docker-entrypoint-initdb.d/
```

### 3. Nginx — Custom Reverse Proxy Config

```nginx
upstream vproapp {
  server vproapp:8080;   # Docker DNS resolves service name
}
server {
  listen 80;
  location / {
    proxy_pass http://vproapp;
  }
}
```

## ⚙️ The Ansible Playbook — 7 Tasks, Full Automation

```yaml
- name: Setup Docker and Deploy Vprofile Stack
  hosts: all
  become: true

  tasks:
    1. Update APT cache + install dependencies (curl, ca-certs...)
    2. Add Docker GPG key
    3. Add Docker APT repository
    4. Install docker-ce + docker-ce-cli + containerd + docker-compose-plugin
    5. Add ubuntu user to docker group
    6. Copy docker-stack/ folder to EC2 (/home/ubuntu/docker-stack/)
    7. Run: docker compose -f docker-compose.prod.yml up -d
```

> 🔑 **The inventory.ini is auto-generated by Terraform** — Terraform writes the EC2 public IP directly into the file after provisioning, so Ansible knows exactly where to connect with zero manual input.

## 🚀 How to Deploy

### Prerequisites

```bash
# Local machine needs:
terraform >= 1.0
ansible >= 2.12
aws cli (configured with your credentials)
ssh key pair (docker-key + docker-key.pub in terraform/)
```

### Step 1 — Provision with Terraform

```bash
cd terraform/

# Generate SSH key pair (one time only)
ssh-keygen -t rsa -b 4096 -f docker-key -N ""

# Initialize and deploy
terraform init
terraform apply -auto-approve

# Output: EC2 public IP + auto-generated ansible/inventory.ini
```

### Step 2 — Deploy with Ansible

```bash
cd ../ansible/

# Wait ~30s for EC2 to boot, then:
ansible-playbook -i inventory.ini playbook.yml
```

### Step 3 — Access the Application

```bash
# Get the IP from Terraform output
terraform -chdir=terraform output docker_ip

# Open in browser:
http://<EC2-PUBLIC-IP>
```

### Step 4 — Cleanup (Don't Forget 💸)

```bash
cd terraform/
terraform destroy -auto-approve
```

## ⚔️ Battles Fought & Engineering Decisions Made

> *Real engineering is debugging the gap between what you think should work and what actually runs in production. These are the four walls we hit — and broke through.*

---

### 🕵️ Battle 1 — The Build vs. Run Trap & Stale Credentials

**The Symptom:** The application was running, but login kept failing. The database was rejecting the password defined in our `.env` file and accepting only the hardcoded `root` password from the instructor's old image.

**Root Cause Diagnosed:** The `docker-compose.yml` had both a `build:` directive and an `image:` directive on the same service. Since the EC2 server had no source code, Docker couldn't execute the build — so it silently fell back to pulling the old pre-built image from Docker Hub. That image had `MYSQL_ROOT_PASSWORD=root` baked in at build time, overriding our runtime environment variables entirely.

**The Architectural Fix — The Two-File Strategy:**

Instead of patching the existing file, a dedicated production-only file `docker-compose.prod.yml` was created with the `build:` directives completely removed. This file references only pre-built images under the `amrmamer/` namespace — images that we built and pushed ourselves with the correct configuration. Ansible was then configured to deploy using the `-f` flag to target this file explicitly:

```bash
docker compose -f docker-compose.prod.yml up -d
```

This also means **the production server ships zero source code** — only the compose file and the `.env` — which is correct security posture.

---

### 🏎️ Battle 2 — The Container Race Condition

**The Symptom:** Tomcat kept crashing on startup with database connection errors. The stack would partially come up, with Tomcat dying repeatedly while MySQL and RabbitMQ were still initializing.

**The Naive Fix (Rejected):** Adding `restart: always` to Tomcat. This would make it crash, restart, crash, restart in a tight loop — consuming CPU, polluting logs, and masking the real problem.

**The Professional Fix — Dependency-Aware Healthchecks:**

Each backing service received a meaningful healthcheck that actually probes its readiness (not just process existence). Tomcat's `depends_on` was then upgraded from a simple `service_started` condition to `service_healthy`:

```yaml
vproapp:
  depends_on:
    vprodb:
      condition: service_healthy    # Waits for mysqladmin ping ✅
    vprocache01:
      condition: service_healthy    # Waits for TCP port 11211 ✅
    vpromq01:
      condition: service_healthy    # Waits for rabbitmq-diagnostics ping ✅
```

The same pattern was applied to Nginx — it now waits for Tomcat to be healthy before accepting public traffic, eliminating the `502 Bad Gateway` window during startup.

---

### 🩺 Battle 3 — The Blind Doctor (Memcached Healthcheck Failure)

**The Symptom:** `docker compose ps` showed `vprocache01` as `unhealthy`, blocking the entire dependency chain. Yet the Memcached process itself was running perfectly.

**The Debugging Move — `docker inspect` Over Guessing:**

```bash
docker inspect vprocache01 --format='{{json .State.Health}}'
```

The output revealed the actual error: `nc: not found`. The healthcheck was using `netcat` (`nc`) to probe port 11211, but the official `memcached` image is a minimal Alpine-based image with no `nc` binary installed.

**The Elegant Fix — Pure Bash TCP Probe:**

Replaced the external binary dependency with a bash built-in that every Linux system has:

```yaml
healthcheck:
  test: ["CMD-SHELL", "bash -c '</dev/tcp/127.0.0.1/11211' || exit 1"]
```

This performs a TCP connection attempt using bash's `/dev/tcp` pseudo-device — no external tools, no package installation, works on every minimal image. The healthcheck turned green immediately.

---

### 🐍 Battle 4 — Environment Issues (Fedora + Python Libraries)

**The Symptom:** While generating the architecture diagram script in Python, a `SyntaxError` appeared, followed by a `ModuleNotFoundError` for the `graphviz` binary even after `pip install graphviz`.

**Root Cause:** The `graphviz` Python package is only a binding — it requires the actual Graphviz rendering engine (`dot`, `neato`, etc.) to be installed at the OS level as a separate binary. On Fedora, the package manager is `dnf`, not `apt`:

```bash
# Wrong (Debian/Ubuntu only):
sudo apt install graphviz

# Correct for Fedora:
sudo dnf install graphviz
```

Additionally, the diagram code was cleaned up to remove deprecated class usage from older `diagrams` library versions, producing a clean architecture diagram with real AWS logos and proper dark-theme styling.

## 🔐 Security Architecture

### What's Exposed vs. What's Isolated

```
PUBLIC (reachable from internet):
  ✅ Port 80   → vproweb (Nginx)        — HTTP traffic only
  ✅ Port 22   → EC2                    — SSH, restricted to MY IP only (/32)

INTERNAL ONLY (Docker backend_net — no host binding needed):
  🔒 Port 3306  → vprodb (MySQL)        — Only vproapp can reach it
  🔒 Port 5672  → vpromq01 (RabbitMQ)  — Only vproapp can reach it
  🔒 Port 11211 → vprocache01           — Only vproapp can reach it
  🔒 Port 15672 → RabbitMQ Management  — Mapped to host for admin use

NETWORK DESIGN:
  frontend_net → bridge (vproapp ↔ vproweb communication)
  backend_net  → internal: true (no outbound internet from DB layer)
```
